# 文档四：并发修改问题、高并发、多线程 —— 这些跟你项目有什么关系？

## 你的场景描述

> "我正在修改部门的某一条数据，同时另外一个页面也在修改同一条数据，这咋办？"

这是一个非常好的问题。这个现象叫做**并发修改冲突**，属于数据库并发控制的范畴。

---

## 第一个问题：这是高并发吗？

**要分情况说：**

### 场景一：2 个人同时操作（你描述的场景）

2 个用户同时修改同一条记录——这不算"高并发"，但会出现**数据覆盖问题（Lost Update，丢失更新）**。

**具体发生了什么：**

```
时间线
─────────────────────────────────────────────────────────────
T1: 用户A 读取 id=1 的部门，name = "技术部"
T2: 用户B 读取 id=1 的部门，name = "技术部"（他也读到了旧值）
T3: 用户A 把 name 改成 "研发部"，提交 → 数据库现在是 "研发部"
T4: 用户B 把 name 改成 "产品部"，提交 → 数据库现在是 "产品部"
─────────────────────────────────────────────────────────────
结果：用户A 的修改被用户B 的修改覆盖了，用户A 不知道，
      他以为自己改成功了，但数据库里已经不是他改的值了。
```

这就是"丢失更新"问题。

### 场景二：每秒几千个请求打过来

这才叫"高并发"。比如秒杀活动、热门商品抢购。你的部门管理系统几乎不会遇到这种场景，但这个概念要知道。

---

## 第二个问题：怎么解决并发修改冲突？

### 方法一：乐观锁（推荐，适合你这种场景）

**思路：** 不加锁，但在提交时检查"我读的时候是什么版本，现在还是这个版本吗？"

**实现：** 在 `dept` 表加一个 `version` 字段（整数）：

```sql
ALTER TABLE dept ADD COLUMN version INT DEFAULT 0;
```

```java
// 修改时，SQL 加上版本号判断
@Update("update dept set name=#{name}, update_time=#{updateTime}, version=version+1 " +
        "where id=#{id} AND version=#{version}")
//                                       ↑ 关键：只有版本号匹配时才更新
int update(Dept dept);
```

```
时间线（用乐观锁之后）
──────────────────────────────────────────────────────────────
T1: 用户A 读取 id=1，name="技术部"，version=5
T2: 用户B 读取 id=1，name="技术部"，version=5（同样读到 version=5）
T3: 用户A 提交修改，SQL: UPDATE dept SET name='研发部', version=6 WHERE id=1 AND version=5
    → 成功！version 从 5 变成 6
T4: 用户B 提交修改，SQL: UPDATE dept SET name='产品部', version=6 WHERE id=1 AND version=5
    → 失败！因为现在数据库里 version 已经是 6 了，不等于 5
    → 返回影响行数 = 0
    → 告诉用户B："数据已被他人修改，请刷新后重试"
──────────────────────────────────────────────────────────────
```

这样就保护了数据安全，没有一个用户的修改会被悄悄覆盖。

### 方法二：悲观锁（不推荐，了解即可）

**思路：** 你查数据的时候就直接加数据库锁，别人查这条数据都得等你提交。

```sql
-- 查的时候加锁
SELECT * FROM dept WHERE id = 1 FOR UPDATE;
-- 其他人查同一条记录会被阻塞，直到你的事务提交
```

缺点：性能差，锁住太久，其他用户体验很差。一般只在特别重要的金融操作中用。

## 第三个问题：多线程跟你的项目有关系吗？

**答案：有关系，但你不需要自己写多线程代码。**

### Tomcat 的多线程

你的 Spring Boot 项目内嵌了 Tomcat 服务器。当多个用户同时发来请求时，Tomcat 会为**每个请求分配一个线程**来处理。

```
用户A 的请求 → 线程1 → 你的 Controller → Service → Mapper → 数据库
用户B 的请求 → 线程2 → 你的 Controller → Service → Mapper → 数据库
（线程1 和线程2 同时在跑，互不影响）
```

**所以：你的代码会被多个线程同时执行，但你不需要自己创建线程。**

### 为什么你要关心这件事？

因为它带来了一个问题：**线程安全**。

```java
// 危险示例（你的项目里没有，但要知道）
@Service
public class DeptServiceImpl {
    private int count = 0;    // 类的成员变量，所有线程共享！

    public void add(Dept dept) {
        count++;    // 多线程同时执行 count++，结果不可预测（竞态条件）
        deptMapper.add(dept);
    }
}
```

**你的项目是安全的**，因为：
- `DeptServiceImpl` 里没有状态变量（成员变量里只有 `deptMapper`，它是无状态的）
- 每个方法的局部变量（如 `LocalDateTime.now()`）是线程私有的，不共享

### 总结：多线程你现在需要知道的

| 概念 | 与你项目的关系 |
|------|-------------|
| Tomcat 多线程处理请求 | 自动的，Tomcat 帮你做 |
| Spring Bean 是单例 | `@Service` 的类只有一个实例，被所有线程共用 |
| 单例 Bean 要无状态 | 不要在 Service 里存实例变量（状态）|
| 局部变量是线程安全的 | 方法内的变量每个线程各自有一份，安全 |
| 你需要自己写多线程吗 | 目前不需要，这是进阶内容 |

## 第四个问题：数据库事务跟并发有什么关系？

你目前的代码**没有加事务**。事务保证"一组操作要么全成功，要么全失败"。

```java
// 如果有个操作需要：1.删除部门 2.同时删除该部门的所有员工
// 如果 1 成功了，2 失败了，数据就不一致了

@Transactional   // 加上这个注解，Spring 会帮你管理事务
public void deleteWithEmployees(Integer deptId) {
    deptMapper.deleteById(deptId);
    // 假设这里出了异常
    employeeMapper.deleteByDeptId(deptId);
    // 如果这里失败，上面的 deleteById 也会回滚
}
```

**你现在每个操作只有一条 SQL，所以暂时不需要事务。** 等后面有了多表操作，再加 `@Transactional`。

